# 階層式設計文件 (Hierarchical Design Document)

> 本文件描述專案從願景到可執行任務的完整分解結構。
> **規範：** 每個最小任務 (T-XXX) 約需 10 個 Commit 完成。
> **關聯目標：** G-001 (核心 MVP 開發)

---

## 層次說明

```
願景 (Vision)
└── 大目標 G-001
    ├── 模組 M-001
    │   ├── 任務 T-010  (~10 commits)
    │   ├── 任務 T-011  (~10 commits)
    │   └── ...
    └── 模組 M-00X
        └── ...
```

| 層次 | 代號 | 說明 |
|------|------|------|
| 0 | 願景 | 產品核心價值與方向 |
| 1 | G-XXX | 大目標，對應 PROJECT_GOALS.md |
| 2 | M-XXX | 功能模組，代表一個完整的系統 |
| 3 | T-XXX | 可執行任務，約 10 個 commit 完成 |

---

## 願景 (Vision)

**凡人修仙傳：放置篇** — 讓玩家在 120 分鐘內體驗從凡人到成仙的完整旅程，透過多樣的屬性、事件選擇與隨機性，實現 64 種不同的結局。

## M-001：核心遊戲循環強化

> **目標：** 使基礎點擊 → 產出 → 建設的循環更豐富且有層次感。

**目的：** 在現有 5 個設施基礎上，增加多樣性與解鎖層次。

8. 整合解鎖條件與境界狀態

## M-004：事件系統

> **目標：** 透過隨機事件與玩家選擇，影響屬性走向與最終結局。

### T-040：事件資料結構設計

**目的：** 定義完整的事件 / 選擇 / 結果資料模型，並產出 30+ 基礎事件。

**資料結構：**
```typescript
interface GameEvent {
  id: string;
  title: string;
  description: string;
  triggerType: 'time' | 'realm' | 'random';
  triggerCondition: (state: GameState) => boolean;
  choices: EventChoice[];
}

interface EventChoice {
  id: string;
  text: string;
  result: EventResult;
}

interface EventResult {
  description: string;
  attributeChanges?: Partial<PlayerAttributes>;
  qiChange?: number;
  flags?: string[];   // 用於結局觸發的標記
}
```

**預期 Commit 拆解（共 ~10）：**
1. 定義 GameEvent / EventChoice / EventResult interfaces
2. 設計 10 個練氣期 / 筑基期的事件資料
3. 設計 10 個金丹期 / 元嬰期的事件資料
4. 設計 10 個化神期 / 大乘期的事件資料（含結局觸發類型）
5. 事件觸發條件設計（境界門檻 + 屬性門檻）
6. EventFlag 常數列表（結局觸發用標記）
7. 在 GameState 加入 eventHistory 與 flags 欄位
8. 事件資料驗證（確保每事件都有 triggerCondition）
9. 事件資料匯出（統一從 data/events.ts 引入）
10. 整合測試，確認事件資料結構可正確解析

**完成標準：** 30 個以上事件資料完整定義，每個有觸發條件、選項與結果。

### T-051：結局觸發檢測系統

**目的：** 定期掃描 GameState，於條件滿足時觸發結局。

**完成標準：** 滿足條件時正確觸發對應結局，Game Loop 暫停。
